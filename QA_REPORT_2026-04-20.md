**Date:** 20 April 2026
**Tester:** Nate (nfish82@hotmail.com)
**Environment:** https://calcheq.com (production) — Railway-hosted, Supabase Postgres
**Account used for testing:** nfish82@hotmail.com / site "Calcheq" / role admin (pilot, trial ends 19 May 2026)
**Test data:** all 7 QA instruments soft-deleted (status = decommissioned). 3 attached calibrations retained against decommissioned instrument (no DELETE endpoint exposed — see minor bug). Test document hard-deleted.

---

## Executive Summary

Calcheq is functional end-to-end for a single-user happy path, but it ships with three critical bugs that silently corrupt its core value proposition (compliance reporting, tenant isolation, and SEO discoverability). Fixing these three in order is the highest-leverage change before onboarding the first paying customer:

1. **Tenant isolation is broken.** A globally unique `instruments.tag_number` constraint means one customer can block another from registering a common tag like "PT-101". This also enables cross-tenant enumeration. (CRITICAL-1)
2. **Compliance numbers lie.** `instruments.last_calibration_date` is set to the *last submitted* calibration rather than the *most recent by date*. A FAIL entered today can be silently overridden by an older PASS submitted afterwards. Dashboard, Reports, Smart Diagnostics, badges — all read from this stale field. (CRITICAL-2)
3. **Every page is effectively invisible to Google.** The canonical URL on every marketing page resolves to the homepage, so the Resources blog strategy contributes zero SEO signal. (CRITICAL-3)

Below these, a cluster of major bugs share a single root cause: **API response-shape mismatch** (frontend reads `response.items` where backend returns `{total, results}`). Fixing the shape contract across Documents, Reports, and Diagnostics should knock out at least four "feature silently shows empty" bugs in one change.

The smaller polish list is long but not blocking. Brand casing, inline date rendering, and trial-copy inconsistencies are mostly 15-minute fixes that would materially lift perceived quality.

**Overall readiness:** safe for a single hand-held pilot; not safe for self-serve sign-ups or any customer who imports historical data.

## Critical

### CRITICAL-1 — `tag_number` uniqueness is GLOBAL across tenants
- **Path:** POST /api/instruments
- **Steps:** Signed in as site "Calcheq" (0 instruments). POST body `{"tag_number":"TEST-QA-001", ...}`
- **Observed:** 409 Conflict, body `{"detail":"An instrument with tag_number 'TEST-QA-001' already exists"}` — yet `GET /api/instruments` for my site returns `total: 0`. Additional tags `PT-101` also 409 while `TT-100`, `FT-200` returned 201 — confirms the conflict is against rows in other tenants.
- **Expected:** Uniqueness should be composite `(created_by, tag_number)` so each tenant owns their own tag namespace.
- **Impact:**
  - Contradicts CLAUDE.md's stated architecture ("Site isolation: always filter by created_by. Never expose cross-site data").
  - Enables cross-tenant enumeration: any authenticated user can probe which tags exist on other sites by attempting inserts.
  - Every new customer importing their tag register (PT-101, TT-100, FT-201 are extremely common in industry) will hit unpredictable 409s.
- **Fix:** Change the UNIQUE index on `instruments.tag_number` to a composite `(created_by, tag_number)`. Update the uniqueness pre-check in `routes/instruments.py` to scope by `created_by` as well. Also applies to PUT (see Major-7).
- **Severity:** Critical — data-integrity + security-perception + customer-onboarding blocker.

### CRITICAL-2 — `instrument.last_calibration_date` uses "last submitted", not "most recent by date"
- **Path:** POST /api/calibrations (any approval path); visible on GET /api/instruments/:id
- **Steps:** Submitted three calibrations against PT-1001 in this order: (1) today (2026-04-20) FAIL @ 1.20% err, (2) 90 days ago (2026-01-20) PASS @ 0.80%, (3) 180 days ago (2025-10-22) PASS @ 0.30%.
- **Observed:** `last_calibration_date = "2025-10-22"`, `last_calibration_result = "pass"`, `calibration_due_date = "2026-10-22"`. Detail header shows a green "CURRENT" badge. The FAIL from today is hidden.
- **Expected:** `last_calibration_date` should be `MAX(calibration_date)` across approved records. A FAIL today should surface as the current result and push the due date to 2027-04-20.
- **Impact:**
  - Compliance reporting is corrupted — a plant with 50 failed calibrations this month can show "100% compliance" if someone later submits an old backdated pass.
  - Due-date scheduling is broken — next due date depends on submission order, not actual dates.
  - CSV bulk import is unsafe — importing historical records in any order other than oldest→newest corrupts every instrument's last status.
  - Status badges lie: "CURRENT" shown on instrument whose most recent record is a FAIL.
  - Cascades into Dashboard Drift Alerts (0/0 shown), Dashboard Compliance Rate (100.0% with a live FAIL), Instrument Health Overview (all "Current"), Smart Diagnostics Recommendations (see Major-12).
- **Fix:** In the approve/submit handler in `calibrations.py`, only update `instrument.last_calibration_*` if `record.calibration_date >= instrument.last_calibration_date`. Recompute `calibration_due_date = last_calibration_date + calibration_interval_days`. Add a defensive one-off migration that recomputes these fields for all existing instruments from `MAX(calibration_records.calibration_date)`.
- **Severity:** Critical — silently corrupts the core metric the product is sold on.

### CRITICAL-3 — Every page declares the homepage as its canonical URL (SEO)
- **Path:** View source of any marketing page (e.g. https://calcheq.com/pricing, https://calcheq.com/blog/pressure-transmitter-procedure)
- **Observed:**
  - `<link rel="canonical" href="https://calcheq.com/">` (the homepage, not the current page)
  - `<meta property="og:url" content="https://calcheq.com">` (same)
- **Expected:** canonical + og:url should resolve to the current page's URL (e.g. `/pricing`, `/blog/<slug>`).
- **Impact:** Google consolidates every page's SEO signal to the homepage and does not index individual articles, pricing, FAQ, etc. The entire Resources blog strategy contributes zero organic search value. sitemap.xml is correct but Google trusts canonical tags over sitemaps.
- **Fix:** In `index.html` drop hardcoded canonical/og:url, then set them per-route via React Helmet (or similar) based on `location.pathname`. Verify by re-viewing source on three different routes.
- **Severity:** Critical (SEO) — every dollar spent on blog content is wasted until this is fixed.

## Major

### Major-1 — Sign-in step 1 "Your company" doesn't bind user to that site
- **Path:** /auth/signin
- **Steps:** Step 1: typed "IXOM" as company, continued. Step 2 copy: "Enter your credentials for IXOM". Signed in with my real (non-IXOM) credentials.
- **Observed:** Login succeeded, user landed on "Calcheq" site (not IXOM). No error, no warning.
- **Expected:** Either reject "This account does not belong to IXOM", or drop the step entirely.
- **Impact:** Misleading UX + a trust-signal failure. Not a privilege escalation (JWT is correctly scoped to real site) but it invites the question. Users will assume the typed company is checked.
- **Fix:** In `/api/auth/me` compare the typed site name to the JWT's site and return 403 if they differ. Or simplify signin to email-only.
- **Severity:** Major — quiet UX failure + perception risk.

### Major-2 — Dashboard fires duplicate API calls on every load
- **Path:** /app (Dashboard)
- **Observed in Network tab per dashboard load:**
  - `/api/dashboard/alerts` × 3 (one unfiltered, two with `?site=Calcheq`)
  - `/api/dashboard/stats` × 2
  - `/api/dashboard/compliance-by-area` × 2
  - `/api/dashboard/upcoming?days=30` × 2
  - `/api/calibrations?record_status=submitted&limit=1` × 3
  - `/api/instruments?last_calibration_result=marginal&status=active&limit=1` × 2
- **Impact:** Wasted server work, slower dashboard, noisy logs. Mixed pattern (some with `?site=X`, some without) suggests the site filter is bolted on inconsistently — a possible source of cross-tenant bugs to audit.
- **Fix:** Consolidate dashboard fetches into a single `useEffect` in `Dashboard.jsx`. Drop the redundant `?site=` query param (the JWT already identifies the site server-side).
- **Severity:** Major — perf + architectural smell.

### Major-3 — Dashboard Drift Alerts card ignores "exceeded" drift status
- **Path:** /app (Dashboard → Drift Alerts card)
- **Observed:** After 3 PT-1001 calibrations where the drift API returns `drift_status: "exceeded"` (rate 1.82%/year, current 1.20% > 1.00% tol), the card shows "0 / No instruments showing drift trends". Smart Analytics on the instrument detail correctly shows "Exceeded — Immediate Action Required".
- **Expected:** Exceeded drift should be the most prominent thing on the dashboard.
- **Fix:** Dashboard drift query should include `drift_status IN ('marginal','exceeded')`. Surface exceeded in red above marginal.
- **Severity:** Major — plant managers miss drift that has already broken tolerance.

### Major-4 — Smart Diagnostics "Drift Alerts" tab also ignores "exceeded"
- **Path:** /app/diagnostics?tab=drift
- **Observed:** Copy says "instruments with drift trend". Empty-state reads "No instruments showing a marginal drift trend right now." PT-1001 (drift_status = exceeded) is absent.
- **Impact:** Same class of bug as Major-3 — between dashboard + this tab + Recommendations, there is literally no screen in the app that surfaces an instrument whose drift has already broken tolerance.
- **Fix:** Include both `marginal` and `exceeded`; render exceeded with an escalated colour.
- **Severity:** Major.

### Major-5 — Smart Diagnostics "Recommendations" shows "All clear" despite FAIL today + exceeded drift
- **Path:** /app/diagnostics (default tab)
- **Observed:** "No recommendations at this time. Your calibration program is in good shape." — while PT-1001 has as_found FAIL today and drift_status = exceeded.
- **Likely root cause:** Recommendations engine reads `instruments.last_calibration_result` which is stale due to CRITICAL-2 (still shows "pass").
- **Fix:** Also cross-check against the most recent approved calibration record and live drift_status, not just the denormalised snapshot. Defence-in-depth on top of fixing CRITICAL-2.
- **Severity:** Major (downstream of CRITICAL-2 but worth fixing independently).

### Major-6 — Smart Diagnostics "Repeat Failures" shows instruments with only 1 failure
- **Path:** /app/diagnostics?tab=failures
- **Observed:** Page copy says "Instruments with 2 or more as-found failures in the last 12 months." Table lists PT-1001 with "Failures: 1".
- **Impact:** Either the threshold is `>= 1` (bug) or the description is wrong. Overstates the bad-actor list and undermines trust in diagnostics.
- **Fix:** Verify backend query uses `>= 2`; otherwise correct the description.
- **Severity:** Major.

### Major-7 — Edit instrument form silently swallows 409 on save
- **Path:** /app/instruments/:id/edit
- **Steps:** Changed description → clicked "Save Changes".
- **Observed:** PUT /api/instruments/:id returned 409 (from the same global tag uniqueness, Major-7b). Page stayed on /edit. No toast, no banner, no inline error. Data NOT persisted — verified by re-reading /api/instruments/:id afterwards.
- **Impact:** Silent data loss on any instrument whose tag collides with another tenant. User has no way to diagnose without Dev Tools.
- **Fix:** (a) Backend: tenant-scope the PUT uniqueness check (same fix as CRITICAL-1 + `WHERE id != :self_id`). (b) Frontend: `InstrumentForm` submit handler must surface API errors as a toast/inline banner — the error branch is swallowing rejections from api.js.
- **Severity:** Major — silent data loss.

### Major-8 — Three different renderings of NULL criticality across the app
- **Path:** /app/instruments + /app/instruments/:id
- **Observed for an instrument with `criticality = null`:**
  - List view → grey "Non-Critical" badge
  - Detail header badge → "Non-Critical"
  - Detail Overview tab → "CRITICALITY: —" (em-dash)
  - Detail Technical Data tab → "CRITICALITY: —"
  - Edit form dropdown → defaults to "Standard" ("Indication only — no direct control or safety function")
- **Impact:** Opening the edit form and hitting Save silently upgrades criticality from null → "standard". Over time this corrupts the risk register. List view also misrepresents risk.
- **Fix:** Treat null as a first-class value. Render as "Unclassified" (amber badge) everywhere. Edit form placeholder "Select…" when null; write the value only if user explicitly picks one.
- **Severity:** Major — quiet data corruption.

### Major-9 — Documents → Upload modal "Link to Instruments" says "No active instruments found" when there are 7
- **Path:** /app/documents → "+ Upload Document" modal
- **Observed:** Modal shows empty state "No active instruments found" while GET /api/instruments?status=active&limit=500 returns 200 with `total: 7`.
- **Likely cause:** Component reads `response.items` (or `response.data`) while API returns `{total, results}`.
- **Impact:** Core feature broken — users cannot link documents to instruments on this site.
- **Fix:** In `Documents.jsx` modal read `response.results`; add tests.
- **Severity:** Major.

### Major-10 — Reports → Failed Report shows "0 of 0 failed calibrations" when API returns 1
- **Path:** /app/reports → "Failed Report"
- **Observed:** GET /api/calibrations?result=fail&date_from=2026-03-20&date_to=2026-04-20&limit=500 returns 200 with `total: 1`. UI displays "0 of 0".
- **Root cause:** Same `response.items` vs `{total, results}` shape mismatch.
- **Impact:** Compliance reporting broken. Combined with CRITICAL-2 and Major-3/4/5, no screen correctly surfaces today's FAIL.
- **Fix:** Update `Reports.jsx` accessors to `response.results`.
- **Severity:** Major.

### Major-11 — DELETE /api/documents/:id returns 503 (but document is deleted)
- **Path:** /app/documents → Delete on any document
- **Observed:** DELETE returns HTTP 503 Service Unavailable. GET /api/documents afterwards shows the row IS gone. UI hides the row without showing an error.
- **Impact:** Non-deterministic error-code on a destructive action. Could signal a post-commit side-effect failure (e.g. Resend email on delete). Users can't rely on the response to confirm.
- **Fix:** Investigate logs — likely a post-commit exception. Wrap side effects in try/except so the response can return 204.
- **Severity:** Major.

### Major-12 — Supabase JWT refresh fails repeatedly in production
- **Console evidence:** 9× `TypeError: Failed to fetch` from `FB._refreshAccessToken` / `FB._callRefreshToken` over a 30-second browsing window. All originate inside the Supabase auth client in `index-DrW0sOFT.js`, not app code.
- **User-visible consequences:** Documents upload modal shows 2× 401 bursts before a 200; correlates with data-not-loading bugs seen elsewhere.
- **Likely causes:** Supabase JWKS endpoint intermittency, CSP/CORS blocking the refresh call, or refresh-interval races against other fetches hitting rate limits.
- **Impact:** Compounds every other "data didn't load" bug.
- **Fix:** Check Supabase project health dashboard; inspect CSP for calcheq.com and ensure the auth origin is whitelisted; add a client-side retry with jitter.
- **Severity:** Major.

### Major-13 — 404/unknown /app/* routes silently redirect to marketing homepage
- **Path:** https://calcheq.com/app/this-page-definitely-does-not-exist (while authenticated)
- **Observed:** Full redirect to marketing landing at `/`. Sidebar disappears, looks like the user has been signed out.
- **Fix:** Add a catch-all `<Route path="/app/*" element={<NotFound/>}>` inside the AuthGuard layout with a "Go to Dashboard" CTA.
- **Severity:** Major — real UX regression. Renamed routes silently break bookmarks.

### Major-14 — App shell is not mobile-responsive
- **Path:** Entire /app/* at <768px viewport widths
- **Observed:** Sidebar `<aside>` keeps `display: flex` at every breakpoint. No `md:hidden` / `md:block` classes on the nav. No hamburger/menu toggle button. At ~767px the 248px sidebar already eats 32% of width; at a real phone width (390px) it would take 64%. Tables rely on `overflow-x-auto` so the user must two-dimension-scroll.
- **Impact:** Blocks any field technician using the app on-site. Marketing copy frequently implies mobile-ready but the app isn't.
- **Fix:** Rework `Layout.jsx` — hide sidebar at `<md`, show a hamburger that opens a drawer; audit grid layouts for `md:grid-cols-*` fallbacks.
- **Severity:** Major.

## Minor

### Minor-1 — Compliance Rate card shows "0.0%" and red "Below 95% target" when there are 0 instruments
- Dashboard with zero instruments shouldn't imply a failing state.
- Fix: render "—" / "N/A" / "Add instruments to see compliance" on empty sites.

### Minor-2 — GET /api/instruments?status=all → HTTP 500
- Any non-enum value on `status` returns 500 (generic "Internal Server Error"). Observed 2× during normal navigation, so the frontend occasionally sends it (probably from a status filter dropdown when user picks "All").
- Fix: accept `all` / empty as "no filter", or reject invalid enum with 400 rather than 500.

### Minor-3 — DELETE /api/instruments/:id is actually a soft-delete but returns 204 with no hint
- DELETE returns 204 empty body. GET after delete still returns 200 with `status: "decommissioned"`. Reasonable for calibration history, but undocumented and confusing to API consumers.
- Fix: return 200 with the updated record (status=decommissioned), or rename route to `POST /api/instruments/:id/decommission` and make DELETE return 405. Or add `?hard=true` for genuine hard delete.

### Minor-4 — No DELETE endpoint for calibration records (405)
- Admin has no UI path or API path to remove a wrongly-entered calibration. If records are immutable by design (audit integrity) that's fine — but provide a "correct" / "void" workflow with a mandatory note.

## Polish

### Polish-1 — Brand casing inconsistency: "Calcheq" vs "CalCheq"
- Tab title reads "Calcheq"; logo/nav/body copy reads "CalCheq"; sign-in page says "New to CalCheq?". Pick one and apply globally.

### Polish-2 — Two different `<title>` tags across app
- Marketing: "CalCheq — Calibration Management for Australian Processing Plants"
- App (post-navigation): "Calcheq — Instrument Calibration Management for Process Industries"
- Drops the Australian positioning on app pages. Align to a single canonical title.

### Polish-3 — Planner "+ Add" doesn't refresh queue panel — user has to click Refresh
- POST /api/queue succeeds (201). Server state is correct. UI still shows "Queue is empty" until Refresh is clicked. Fix: optimistic update or re-fetch on success.

### Polish-4 — Documents delete confirmation sentence breaks with title on new line
- Renders "Are you sure you want to delete ? This action cannot be undone." with the title below as a separate block. Fix: inline the title inside the sentence: "Are you sure you want to delete 'QA-TEST-DOC-20260420'?"

### Polish-5 — Documents list table "Uploaded By" column renders empty
- API returns `uploaded_by: "Nathan Fish"`. Table cell is blank. Likely wrong field name in `Documents.jsx` (reading `uploader_name` / `uploaded_by_name`). Fix to `uploaded_by`.

### Polish-6 — Billing fine print claims "30-day free trial" (inconsistent with 14-day self-serve)
- Settings → Billing → "Choose a Plan" picker shows hardcoded "30-day free trial included". Self-serve signup is documented in CLAUDE.md as 14-day; pilot/contact flow is 30-day. A self-serve user seeing this on day 15 will feel misled.
- Fix: compute trial length from `trial_ends_at - now`, or render the actual end date inline.

### Polish-7 — Trial length is described inconsistently across marketing + app
- Marketing site consistently says 30-day (FAQ, Contact H1 "Start Your Free 30-Day Pilot", Pricing CTA, hero CTA). CLAUDE.md says self-serve = 14 days, pilot = 30 days. Billing copy hardcoded to 30-day.
- Fix decision needed: unify on 30 days (simplest) or make self-serve copy say 14-day explicitly.

### Polish-8 — Billing panel sentence: trial-end date renders as a separate block
- "Your free trial ends on . Choose a plan below..." — the date 19 May 2026 is a sibling element, so the sentence reads broken. Same root cause as Polish-4.

### Polish-9 — Resources cards link to /blog/:slug, not canonical /resources/:slug
- CLAUDE.md describes /resources/:slug as canonical; /blog/:slug is legacy. Resources page cards use /blog/<slug>. Both URLs serve the same content with no redirect — duplicate-content signal to Google (compounded by CRITICAL-3).
- Fix: switch cards to /resources/:slug; add 301 redirect from /blog/:slug.

## Observations (not bugs)

- **IXOM account is empty.** Sidebar + header + dashboard all show "Calcheq" (the user's own site, which holds no real data). CLAUDE.md guidance to treat IXOM as "real pilot — do not delete" still noted; all test activity was on Calcheq site.
- **Response shape contract.** The single highest-leverage refactor below the three critical bugs is to unify the API response shape (`{total, results}`) and fix all frontend accessors in one change — that alone resolves Major-9 and Major-10 and prevents the same bug recurring.
- **Pattern: stale denormalised fields.** CRITICAL-2 is the clearest instance but the pattern (reading from `instruments.last_*` fields instead of aggregating live from `calibration_records`) shows up in Dashboard, Smart Diagnostics Recommendations, and likely Reports. Worth a dedicated audit.
